Competition (Stage 1)

Final stage

Jean runs a small contracting firm, IO.co, staffed by rudimentary androids who cannot think for themselves. Knowing the limitations of their employees, Jean only accepts jobs broken down into repetitive, identical tasks. A recent bout of overconfidence led Jean to take contracts for more work than IO.co can handle. IO.co is even contracted for jobs that can't be started yet! They've stopped taking new work but Jean needs to figure out a schedule for the remaining jobs — one that will keep IO.co's customers happy.

Jean wants to minimize two metrics: 1. The amount of time between when the job can be started and it is completed, to keep the customers happy; and 2. The amount of time between when a job is actually started and it is completed, to cut down on multitasking. To measure this, Jean devised the following cost function for potential schedules:

∑j=1npj(ej−bj)2+(ej−sj)2‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾√∑j=1npj(ej−bj)2+(ej−sj)2
where, for jobs {j1,j2,j3,…,jn}{j1,j2,j3,…,jn}, pjpj is the priority of the job (based on how angry the client will be if it's slow), sjsj is the earliest the job can have been started, bjbj is the time the job was actually started, and ejej is the time the job was completed. Unfortunately for Jean, determining the cost of a schedule is much easier than determining the best schedule, or even a good one. If IO.co is going to stay afloat, they'll need all the help they can get.

Each job is divided into a number of tasks which all take the same amount of time. Each task can be worked on by one android at a time. Instructions to begin tasks may preempt tasks an android is already working on. An incomplete task, if interrupted by assignment of a new task, is abandoned and must be started from nothing when resumed. Multiple tasks from one job can be worked on simultaneously. Each android can work on a single task at a time.

Input

The input is a list of tuples describing the jobs and IO.co's current employees. Each job entry lists the name of the job, the number of tasks in the job, how long each task takes to complete, the earliest time the job can be started, and the job priority. Time is unitless and starts at 00. Tuples for jobs have the format 'job NAME TASKS TASK_TIME CAN_BEGIN PRIORITY'. Tuples for androids have the format 'worker NAME'. It is guaranteed that all names are unique and do not have spaces. Tuples are separated by newlines (ASCII code 10) and fields in tuples are separated by spaces (ASCII code 20).

Output

Display a list of instructions for IO.co's androids, in order by time from earliest to last, indicating the time the instruction should be given to the android, the android it should be given to, and the job they should select a task from. Each instruction should be a tuple with the format 'TIME ANDROID JOB'. Tuples should be separated by newlines (ASCII code 10) and fields within tuples should be separated by spaces (ASCII code 20). If an android does not receive additional instructions they will continue working on new tasks from the same job. If the job an android is working on is completed before the android recieves new instructions, the android will become idle until given new instructions. If instructions are out of chronological order, will leave a job incompleted, or are impossible (e.g. instruct androids to work on a job already completed) you will receive no credit.

Sample input & output

Input

job count_doodads 119 11 0 1
job stamp_whatsits 264 5 22 1
worker Android_Joe
worker Android_Mary

Output

0 Android_Joe count_doodads
0 Android_Mary count_doodads
33 Android_Joe stamp_whatsits
1276 Android_Mary stamp_whatsits

The input gives the name of IO.co's two employees and defines two jobs:

  • count_doodads, which has a priority of 1, is divided into 119 tasks each of which takes 11 units of time to complete, and which can be started immediately.
  • stamp_whatsits, which has a priority of 1, is divided into 264 tasks each of which takes 5 units of time to complete, and which can only be started after 22 units of time have passed.

The output instructions see both Android Joe and Android Mary start counting doodads. At time 33, Android Joe is switched to stamping whatsits (having already counted 3 doodads). When, at time 1276, Android Mary finishes counting doodads, she is switched to help Android Joe finish stamping whatsits.

Running your program

Input will be given to your program via stdin (standard input) in ASCII encoding. Your program must write its output in ASCII encoding to stdout (standard output).

Scoring

The overall cost of the schedule your program produces will be calculated with Jean's formula:

∑j=1npj(ej−bj)2+(ej−sj)2‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾√∑j=1npj(ej−bj)2+(ej−sj)2
This cost will be compared to the cost of a naive schedule composed as follows: With jobs sorted by the time they can begin from earliest to latest, each job is assigned in turn to the next available android. That android will work until the job is completed. If there are no available androids, the job will be assigned to the first andoir that becomes idle, and so on until all jobs have been completed or are assigned to an android.

The score for a schedule with cost cxcx is the percentage savings over the cost cncn of naive schedule, 100⋅(1−cx/cn)100⋅(1−cx/cn) or 11, whichever is greater, e.g. if the naive schedule costs 50 and your job costs 20, your score is 100⋅(1−20/50)100⋅(1−20/50), or 6060. Your program will be run 10 times for different sets of jobs and employees, each with up to 1000 jobs, and the scores of each run summed to produce your final score. Each run must complete in 40 seconds.

Note that it is very easy to write a naive scheduler and much harder to write a good one. Even when you have a working solution, it will be a good idea for you to revisit your scheduler and search for ways to improve it.
